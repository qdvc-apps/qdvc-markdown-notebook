# Vibe-coding log

This folder is the vibe-coding transparency record for this codebase. The codebase is
**vibe-coded** — written with substantial help from a large language model — and
the `qdvc` ("quick-and-dirty vibe-coded") prefix in the repository name says so
up front. The files here preserve the LLM interactions behind the code so that
anyone (human or AI) can audit how a given change came to be.

This file was also generated by LLM (Claude, Opus 4.8 High).

This README documents the conventions for that record. It describes *how* the
project is documented, not *what* the code does — for the latter, see the
top-level `README.md` (usage) and `MAINTENANCE.md` (internals).

## What gets logged

Every working session with an LLM that produces or modifies tracked files is
recorded as a transcript in this folder. A session is logged if it changed the
repository, including code, documentation, shell completions, or this log
itself. Throwaway exploratory chats that produced nothing committed do not need
their own file, though it is fine to keep them.

## File naming

One file per day of work:

```
YYYY-MM-DD-claude.md
```

If a single day holds two clearly separate sessions, append a short suffix
(`YYYY-MM-DD-claude-2.md`). Keep the date in ISO order so the
files sort chronologically.

## Transcript structure

Each transcript opens with a level-1 heading and, on the next non-blank line,
the model and reasoning effort used:

```
# YYYY-MM-DD Conversation with Claude

Opus 4.8 High
```

The body alternates between the human's prompt and the model's reply, numbered
sequentially through the file:

```
## 1. User request

...the prompt, verbatim...

## 2. LLM response

...the reply...

## 3. User request

...
```

Number every section in one running sequence (1, 2, 3, 4, …) regardless of
speaker, so each section has a unique reference within the file.

## What to preserve

- **The prompt, verbatim.** Include standing rules and constraints (for example,
  "do not reflow markdown", "all files must end with a blank line"). These
  explain why the output looks the way it does and are the most useful part of
  the record.
- **The model's response**, including any narration of its intermediate steps.
  This may be lightly trimmed for noise, but do not rewrite it to look cleaner
  than it was.
- **Honest endings.** If a session ended incomplete — hit a tool limit, left a
  task half-finished — say so in the transcript rather than tidying it away. The
  log is a record, not a highlight reel.

## Linking transcripts to commits

So that a reader can map a conversation to the lines it changed, record the
resulting commit(s) at the end of the relevant `LLM response` section:

```
_Committed as 12a34bc_
```

Because the commit is usually made *after* the reply, this is added when the
work lands. If a session spans several commits, list them all. This is the one
piece of provenance the transcript cannot capture on its own, so it is worth the
small effort.

## Commit trailers

For machine-readable provenance that shows up in `git log` without opening this
folder, add a co-author trailer to AI-assisted commits:

```
[feat] Data export functionalities

Co-authored-by: Claude <noreply@anthropic.com>
```

GitHub renders `Co-authored-by:` trailers as co-authors, so this makes the AI
contribution discoverable from the history alone. The trailer and the
transcripts complement each other: the trailer marks *that* a commit was
AI-assisted; the transcript shows *how*.

## Model disclosure

The top-level `README.md` names the model used in its Acknowledgements section
and points here. Keep that pointer current if the toolset changes.

## File conventions

Match the rest of the repository: UTF-8, Unix line endings, and a single
trailing newline at end of file. Do not reflow existing transcript text when
appending to a file — add new sections rather than rewrapping old ones.
