### Summary: Deviations of `qdvc-markdown-notebook` from the spec

I've read the app's `README.md`, `MAINTENANCE.md`, and source, and fetched a fresh copy of the spec (latest commit `7cdbc47`, dated 2026-07-14). The app was clearly written against an earlier, simpler version of the spec and predates several structural requirements. The deviations fall into two buckets: **structural/architectural** (large, spec-mandated but unimplemented) and **smaller conformance gaps**.

#### Structural deviations (the big ones)

1. **No GTK 4 front-end (§9, §2).** The app is GTK 3-only. The spec now requires (SHOULD, strongly) a parallel GTK 4 / libadwaita front-end on the same pure core, with a `MAINTENANCE_GTK3_GTK4.md`. The code is architected in anticipation of this (GTK-free core + `gtk3_`-prefixed view modules), and MAINTENANCE.md §7 lists it as a "suggested next feature," but it doesn't exist yet.

2. **Package folder is `qdvcmdnb_lib/`, not `qdvc/` (§2).** The spec mandates the package be named exactly `qdvc/`, with nested `qdvc/gtk3/` and `qdvc/gtk4/` sub-packages and `gtk3_*`/`gtk4_*` module prefixes. Currently everything (core + `gtk3_*`) sits flat inside `qdvcmdnb_lib/`.

3. **No backend dispatcher / `--gtk3`/`--gtk4` flag / `ui_backend` config key (§3).** The entry point just parses a folder argument; there's no toolkit selection logic, since there's only one toolkit.

4. **No application ID (§7.1).** The app sets a prgname (`qdvc-markdown-notebook`) but never defines an `APP_ID` of the form `qdvc.MarkdownNotebook` for the `Gtk.Application`. In fact the app doesn't use `Gtk.Application` at all — the entry point calls `Gtk.main()` with a bare `Gtk.Window` subclass, whereas §8 says `gtk3_app.py` MUST define a `Gtk.Application` (id, `HANDLES_OPEN`, `do_startup`/`do_activate`/`do_open`).

5. **Docs not under `docs/` (§12).** `MAINTENANCE.md` and `README.md` are at the repo root. The spec requires `docs/MAINTENANCE.md` (and `docs/MAINTENANCE_GTK3_GTK4.md` once GTK 4 exists). Also, §12 says MAINTENANCE.md MUST reference the spec by name and URL and MUST NOT duplicate it — the current MAINTENANCE.md is a ~900-line standalone document that doesn't reference the spec at all and restates much shared material.

6. **No shared `SHORTCUTS` table / `ui_prefs.py` (§10).** Shortcuts are wired ad-hoc in the window/menubar code; there's no pure `ui_prefs.py` with a canonical `SHORTCUTS` table. (This is partly moot until GTK 4 exists, but the pure table is still the spec's single source of truth.)

7. **No `platform_utils.py` (§11).** "Show in file browser" is done inline via `Gtk.show_uri_on_window`; there's no pure `platform_utils.py` with `open_with_default_app` / `open_with_text_editor` / `reveal_in_file_manager`.

#### Smaller conformance gaps

8. **Config directory naming (§5).** Config lives at `~/.config/qdvcmdnb/config.yml`. The spec's canonical location is `~/.config/qdvc-<app>/config.yml` — i.e. `qdvc-markdown-notebook/`. (The YAML format itself conforms.)

9. **Config API shape (§5).** `settings.py` uses a typed `Settings` class with explicit `set_*` accessors and a `SCHEMA_VERSION`, whereas the spec asks for a thin `get(key, default)` / `set(key, value)` wrapper over a `DEFAULTS` dict, specifically so no schema migration is needed on a new key. This is a design divergence, though the app's `_extra` forward-compat mechanism achieves a similar robustness goal.

10. **Python version target (§4).** MAINTENANCE.md states "Python 3.6+" and the code uses `.format()`-style patterns; the spec now mandates **Python 3.10+** (union syntax, `list[...]` generics). Not a functional break, but the stated target and idioms lag the spec.

11. **Menu icon widget detail (§8).** The app builds icon menu items via `Gtk.ImageMenuItem` (correct, and explicitly endorsed by §8's deprecation rationale), but the spec also says the app SHOULD call `set_always_show_image(True)` and provide icon-name fallback resolution.
